# CEI Framework — Scenario Test Suite

This directory contains five reference scenarios that exercise the CEI (Centrality-Entropy Index) framework across structurally distinct infrastructure domains. Each scenario is a self-contained dataset with topology, governance policies, and synthetic telemetry. Running the scenarios produces a comparison between a standard threshold-based autoscaler (HPA-style) and the CEI decision pipeline.

## Why Multiple Scenarios

A resource-allocation framework is only as interesting as the range of domains it generalizes across. Rather than demonstrating the framework only on commercial cloud workloads, this suite spans five structurally different environments to show that the underlying analytical principle — measuring centrality, entropy, and governance risk in a single integrated score — produces sensible decisions regardless of domain.

## Running

```bash
# Run a single scenario
python3 runner.py --scenario cloud_microservices

# Run all five
python3 runner.py --all
```

Output includes per-node CEI scores (centrality, entropy, governance risk), HPA baseline decision counts, and CEI decisions with reasoning. Results are written to `results.json` in each scenario directory.

### topology.json

## What the Scenarios Demonstrate

Each scenario exercises specific aspects of the framework:

**Cloud microservices** — oscillation detection on autoscaler hotspots (`catalog-svc`, `order-svc`), idle-but-protected detection for compliance-tiered nodes, structural waste identification in discretionary tier.

**GPU cluster** — heterogeneous accelerator classification, checkpoint-integrity governance blocking during active writes, inference SLO enforcement, data-residency constraint respect.

**Drone swarm** — latency-budgeted allocation, link-redundancy enforcement, degradation detection on individual platforms, hot-standby preservation for mission coordinator.

**Underwater APS** — bandwidth-scarcity-aware allocation, minimum-positioning-sources enforcement for critical UUVs, frequency-band coordination, ocean-noise adaptivity.

**NC3 strategic comms** — correct handling of systems where low utilization is by design, path-diversity enforcement, crypto two-person-integrity preservation, network segmentation awareness.

## Notes

- All scenarios use unclassified, publicly-available architectural concepts. No scenario represents any specific deployed system.
- Telemetry is synthetic, generated with realistic utilization patterns (stable, oscillating, bursty, degrading, idle, saturated).
- The runner uses simplified centrality (PageRank approximation) and Shannon entropy; the production core engine implements additional methods.
- HPA baseline in the runner is deliberately simple (threshold-based) to represent the most common default in Kubernetes deployments.
